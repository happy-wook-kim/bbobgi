# bbobgi (뽑기) 설계 문서

작성일: 2026-07-07

## 1. 개요

커피값 내기처럼 **한 화면에 모여서** 하는 캐주얼한 랜덤 추첨 웹사이트. 한 명을 뽑거나 여러 명에게 결과를 배정하는 두 가지 상황을, 카드·룰렛·사다리 세 가지 연출로 즐길 수 있다. 서버 없이 순수 프론트엔드로 동작한다.

## 2. 결정 사항 요약

| 항목 | 결정 |
|---|---|
| 기술 스택 | Vite + React + TypeScript |
| 실행 환경 | 한 화면에 다같이 (서버·실시간 동기화 없음) |
| 배포 | `vite build` → 정적 파일 → GitHub Pages 또는 Vercel |
| 결과 모드 | ① 한 명 당첨 · ② 각자 배정 (둘 다 지원) |
| 연출 | 🃏 카드 · 🎡 룰렛 · 🪜 사다리 (3종) |
| 참가자 표시 | 랜덤 단어/이모지 배정 |
| 스릴 방식 | 각자 확인 후 숨김 (순서대로 본인 단어 확인) |

## 3. 핵심 개념 — 3축 분리

세 가지가 서로 독립적인 축이며, 조합으로 동작한다.

1. **엔진 (순수 로직)** — 누가 무엇을 뽑았는지 계산한다. 랜덤 공정성을 책임진다.
2. **연출 (애니메이션)** — 엔진이 계산한 결과를 *보여주기만* 한다. 결과를 바꾸지 못한다.
3. **결과 모드** — 뽑기의 목적. 한 명만 뽑을지, 전원에게 배정할지.

엔진과 연출을 분리하는 이유: 랜덤 공정성을 한 곳에서 보장하고, 연출을 나중에 추가하기 쉽게 하기 위함이다.

### 두 축의 조합

| 축 | 선택지 |
|---|---|
| **연출** | 🃏 카드뽑기 · 🎡 룰렛 · 🪜 사다리 |
| **결과 모드** | ① 한 명 당첨(다 냄) · ② 각자 배정(전액/반값/면제…) |

## 4. 사용자 흐름

한 화면(폰/노트북 하나)을 여럿이 돌려보는 것을 전제로 한다.

```
1. 참가자 입력      이름 입력 (예: 철수, 영희, 민수…)
2. 모드 선택        ① 한 명 당첨  /  ② 각자 배정
                    ②를 고르면 결과 항목도 입력 (전액/반값/면제…)
3. 단어 배정 확인    각 참가자에게 랜덤 단어/이모지 배정 →
                    순서대로 본인 단어 확인 후 가리고 다음 사람
4. 연출 선택        카드 / 룰렛 / 사다리
5. 실행 (두근두근)   선택한 연출 애니메이션
6. 결과 공개        당첨자/배정 결과 + "다시하기"
```

## 5. 단어 배정 & 스릴

- 참가자에게 미리 준비된 단어/이모지 풀(🍎사과 🍌바나나 🍇포도 …)에서 **겹치지 않게** 랜덤 배정한다.
- 카드·룰렛에는 이름 대신 **배정된 단어**가 적힌다. 뽑힌 단어의 주인이 결정된다.
- **스릴(각자 확인 후 숨김):** "단어 배정 확인" 단계에서 참가자 목록 순서대로 한 명씩 본인 단어를 확인한다.
  - "○○님 차례 — 탭하면 내 단어 공개" → 단어 표시 → "확인" 누르면 즉시 가려지고 다음 사람으로 넘어간다.
  - 누가 어떤 단어를 받았는지 공개되지 않으므로, 뽑는 순간까지 긴장감이 유지된다.

## 6. 연출별 동작

| | ① 한 명 당첨 | ② 각자 배정 |
|---|---|---|
| 🃏 **카드** | 단어 카드를 엎어놓고 → 한 장 뒤집어 당첨자 공개 | 각자 카드 한 장씩 뽑아 결과 확인 (N개 카드 = N개 결과) |
| 🎡 **룰렛** | 단어들이 파이 조각 → 돌려서 멈춘 조각의 주인 당첨 | 돌릴 때마다 결과를 하나씩 소진하며 순서대로 배정 |
| 🪜 **사다리** | 출구 하나만 "🎯당첨", 나머지 "통과" → 타고 내려가 걸린 사람 | N개 입구 → N개 결과 출구, 전형적 사다리타기 (N:N 매핑) |

모든 연출은 엔진이 미리 계산한 결과를 재생할 뿐이며, 애니메이션이 결과를 결정하지 않는다.

## 7. 아키텍처 / 폴더 구조

```
src/
  engine/
    shuffle.ts        Fisher-Yates 셔플
    draw.ts           draw(mode, participants, outcomes) 순수함수
    tokens.ts         단어/이모지 풀 + 랜덤 배정
  components/
    SetupScreen       참가자 / 모드 / 결과 항목 입력
    TokenReveal       각자 단어 확인 (순서대로, 확인 후 숨김)
    animations/
      CardDraw
      Roulette
      Ladder
    ResultScreen      결과 공개 + 다시하기
  App.tsx             단계 상태머신 (setup → reveal → animate → result)
```

## 8. 데이터 모델 (개략)

```ts
type Participant = { id: string; name: string; token: string }; // token = 배정된 단어/이모지
type Outcome = { id: string; label: string };                   // 모드②의 결과 항목

type DrawResult =
  | { mode: 'single'; winnerId: string }
  | { mode: 'assign'; assignments: Record<string /*participantId*/, string /*outcomeId*/> };
```

## 9. 랜덤 공정성

- 뽑기 로직은 Fisher-Yates 셔플 기반 순수함수로 구현하며, 특정 참가자/결과가 유리하지 않도록 균등 분포를 보장한다.
- 결과는 연출 재생 이전에 확정되고, 연출은 확정된 결과를 바꿀 수 없다.

## 10. 테스트 전략

- 엔진(`draw`, `shuffle`, `tokens`)은 순수함수이므로 Vitest 단위테스트로 검증한다.
  - 모드①: 정확히 한 명의 당첨자가 나오는지.
  - 모드②: 모든 결과가 빠짐없이, 중복 없이 배정되는지.
  - 단어 배정: 참가자 수만큼 겹치지 않게 배정되는지.
- 애니메이션(연출) 컴포넌트는 단위테스트 대상에서 제외한다.

## 11. 배포

- `vite build` 결과물은 정적 파일이므로 GitHub Pages 또는 Vercel에 그대로 배포한다.

## 12. 범위 밖 (YAGNI)

다음은 이번 범위에 포함하지 않는다.

- 서버, 데이터베이스, 실시간 동기화 (각자 기기 접속 방식)
- 사용자 계정 / 로그인
- 결과 이력 영구 저장 (세션 내 "다시하기"만 지원)
- 4번째 이상의 연출
